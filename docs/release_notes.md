# Release Notes

Version: 1.0
Last updated: 2026-05-21
Audience: pilot-to-GA customers

Status: pilot/expansion ready, not GA ready. See `docs/ga_readiness.md`.

## Customer-Visible Behavior

- Operators can manage policy/SOP documents, role packs, missions, guardrail
  quizzes, cohorts, and launches through authenticated application routes.
- Learners can access assignments, complete guardrail quizzes, submit artifacts,
  and view feedback status.
- Managers can review submissions, approve workflow changes, view dashboards,
  and create progress reports.
- Governance workflows now include policy approval, risk taxonomy, control
  lineage, audit export, model registry, feedback quality eval, and human
  sampling/adjudication.
- Reliability and commercial artifacts now include SLOs, load-test harness,
  incident response, migration rehearsal, packaging, ROI calculator, procurement
  packet, implementation success plan, and GA readiness checklist.

## Major Changes

| Area | Change |
|------|--------|
| Product UX | Authenticated operator, learner, and manager HTTP UI surfaces exist for core pilot workflows. |
| Security | OIDC SSO boundary, RBAC permissions matrix, backup/restore/retention controls, and security packet are documented and tested. |

## Migrations

Schema migrations added after the MVP foundation include:

- `0012_document_approval.py` for source document approval state.
- `0013_model_registry.py` for model registry records and feedback version refs.
- `0014_feedback_sampling.py` for feedback sample review and adjudication labels.

Run migrations with:

```bash
alembic upgrade head
```

Before any customer upgrade, follow `docs/migration_rehearsal.md` and create a
fresh backup using `docs/backup_restore.md`.

## Rollback

Rollback rules:
- If application code changes fail but the schema is backward compatible, roll
  back the application image/container first.
- If schema changes are not backward compatible or data mutation is partial, stop
  web and worker processes and restore the latest known-good backup.
- Preserve the failed database, logs, migration output, and audit events for
  incident review.
- Use `docs/incident_response.md` for failed migrations, data leak suspicion, or
  provider degradation.

## Known Limitations

- GA readiness is not yet met.
- Browser automation remains an inherited P2 UX readiness gap.
- Production dashboards, alert wiring, real deployment load tests, and restore
  drill evidence are still required for production-scale claims.
- Paid customer or signed expansion path evidence is not yet recorded.
- SaaS-grade tenant isolation claims require a future ADR and isolation tests.

## Upgrade Impact

Expected customer impact:
- Existing pilot data remains versioned by workspace, cohort, assignment,
  submission, source snapshot, feedback result, and report IDs.
- Newly imported knowledge documents remain pending until human approval.
- Feedback results should include version references when generated by the
  worker after the model registry migration.
- Existing manually seeded feedback rows may use `untracked` version defaults.
- Feedback sampling/adjudication creates new review records without mutating
  original feedback content.
