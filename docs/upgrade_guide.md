# Upgrade Guide

Version: 1.0
Last updated: 2026-05-21
Audience: deployment owner

Use this guide for pilot-to-expansion upgrades. Do not use it to claim GA
readiness; `docs/ga_readiness.md` currently marks the product not ready for GA.

## Before Upgrade

1. Review `docs/release_notes.md`.
2. Confirm target package tier and deployment option.
3. Review open findings in `docs/CODEX_PROMPT.md` and `docs/audit/AUDIT_INDEX.md`.
4. Run the migration rehearsal in `docs/migration_rehearsal.md`.
5. Create a fresh backup according to `docs/backup_restore.md`.
6. Record current application SHA and `alembic current`.

## Upgrade Steps

1. Stop background workers or pause job processing.
2. Deploy the new application image or source revision.
3. Run:

   ```bash
   alembic upgrade head
   ```

4. Start web and worker processes.
5. Run smoke checks:
   - `GET /health`
   - core dashboard/report query
   - retrieval eval in `--no-write` mode if retrieval or document approval changed
   - `.venv/bin/pytest tests/integration/test_migrations.py -q`

## Migrations

Required migrations:
- `0012_document_approval.py`
- `0013_model_registry.py`
- `0014_feedback_sampling.py`

Expected schema changes:
- Source documents have approval status and approver metadata.
- Feedback results have prompt, model, rubric, corpus, and schema version refs.
- Model registry records store durable version metadata.
- Feedback sampling and adjudication tables store review queue and label records.

## Customer-Visible Behavior

After upgrade:
- New and updated source documents require human approval before retrieval use.
- Feedback generated by the worker records model/prompt/rubric/corpus/schema
  versions.
- Operators can use integration surfaces when explicitly configured.
- Managers may see governance/risk/reporting improvements.
- Human feedback sampling and adjudication may be available for Governance Plus
  workflows.

## Rollback

Application rollback:
- Use when schema remains compatible and only application behavior regressed.
- Roll back the app image/source revision and restart web/worker processes.

Database rollback:
- Use when migration partially applied, schema is incompatible, or data
  validation fails.
- Stop web and worker processes.
- Restore the latest known-good backup into a clean database.
- Point the deployment to the restored database.
- Run health, dashboard/report, migration, and retrieval smoke checks.
- Preserve failed database and logs for incident review.

## Validation

Upgrade is accepted only when:
- `alembic current` matches the intended head.
- Health check passes.
- Migration integration test passes.
- Audit event count did not decrease.
- Core pilot path remains usable.
- No raw learner artifact, policy/SOP body text, manager note, email, full name,
  secret, or customer text appears in logs, metrics labels, release notes, or
  upgrade evidence.

## Communication

Tell customers:
- What changed.
- Whether any new approval step is required.
- Whether integrations need new configuration.
- Whether reports, audit exports, or governance surfaces changed.
- What rollback window and support contact apply.

Do not promise guaranteed productivity gains or GA readiness.
